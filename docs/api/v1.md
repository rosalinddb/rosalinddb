# RosalindDB v1 API contract

Authoritative HTTP contract. Implementations are in
`services/control_plane/cp_app.py` (which composes `services/source_registry/main.py`
and `services/query_api/query_proxy.py`) and `services/query_api/` (the Query Data
Plane that backs `/v1/query`).

## Conventions

- **Base URL** (dev): `http://localhost:8080`. The Control Plane is the single
  public origin; `/auth/*`, `/v1/datasets*`, and `/v1/query*` are all served
  from it. Internally the CP proxies `/v1/query` to a private Query Data
  Plane; externally that hop is invisible.
- **Content-Type** is `application/json` unless noted.
- **Authentication** is `Authorization: Bearer <token>` where `<token>` is
  either:
  - A JWT issued by `POST /auth/login` or `POST /auth/signup` (used by the
    dashboard), or
  - An API key of the form `rb_live_<32 url-safe chars>` (used by customer
    code).
  - Both resolve to the same `tenant_id`; downstream handlers don't know or
    care which was used.
  - When `RB_REQUIRE_AUTH` is unset/false (the OSS default), the
    `Authorization` header is ignored entirely and every request resolves to
    a single built-in `default` tenant. The `/auth/*` mutation endpoints
    return `404 auth_disabled` in that mode — see [`auth.md`](./auth.md).
- **Error envelope** for any 4xx/5xx response (built by
  `services/auth/auth.py:install_exception_handlers`):
  ```json
  { "error": { "code": "snake_case_code", "message": "Human-readable explanation", "details": { } } }
  ```
  `details` is optional and endpoint-specific.
- **IDs** are opaque strings. Tenant and API-key ids are
  `ten_<uuid4hex>` / `key_<uuid4hex>`; import-job ids are
  `imp_<uuid4hex>`; query job ids are `job_<uuid4hex>`. Names (dataset
  names, key names) are caller-chosen; constraints documented per endpoint.
- **Timestamps** are ISO 8601 UTC (`"2026-05-14T12:34:56Z"`) except where
  noted (`queries_reset_at` is a `YYYY-MM-DD` date string).
- **Tenant isolation**: every endpoint that operates on tenant data filters
  by the caller's `tenant_id`. Cross-tenant requests return `404 not_found`,
  never `403` — existence is never leaked.

## Auth

The endpoints below are only reachable when `RB_REQUIRE_AUTH=true`. With the
OSS default (`RB_REQUIRE_AUTH` unset/false) they all return
`404 auth_disabled` and there is no per-tenant signup flow — every caller
resolves to the bootstrap `default` tenant without an `Authorization`
header. See [`auth.md`](./auth.md) for the full quickstart.

### `POST /auth/signup`

Create a new tenant + first user. Auto-issues a default API key.

**Request:**
```json
{ "email": "you@example.com", "password": "at-least-8-chars" }
```

**Response 201:**
```json
{
  "token": "eyJhbGc...",
  "tenant": {
    "id": "ten_01H...",
    "email": "you@example.com",
    "plan": "free",
    "created_at": "2026-05-14T12:34:56Z"
  },
  "first_api_key": {
    "id": "key_01H...",
    "key": "rb_live_abc123...",
    "name": "Default",
    "created_at": "2026-05-14T12:34:56Z"
  }
}
```

`first_api_key.key` is the **only** time the raw key is returned.

**Errors:**
- `400 invalid_email` — malformed
- `400 weak_password` — < 8 chars
- `409 email_taken`
- `404 auth_disabled` — `RB_REQUIRE_AUTH` is unset/false

### `POST /auth/login`

**Request:** `{ "email": "...", "password": "..." }`

**Response 200:** Same shape as signup minus `first_api_key`.

**Errors:** `401 invalid_credentials`, `404 auth_disabled`

### `GET /auth/me`

Returns the current tenant. Requires JWT or API key.

**Response 200:**
```json
{
  "tenant": {
    "id": "ten_01H...",
    "email": "you@example.com",
    "plan": "free",
    "created_at": "2026-05-14T12:34:56Z"
  }
}
```

**Errors:** `401 unauthorized` if no/invalid token; `404 auth_disabled` if
auth is off.

### `POST /auth/keys`

Issue a new API key. JWT-only — customer code cannot mint more keys from an
existing API key.

**Request:** `{ "name": "Production server" }` (name 1-64 chars, free-form)

**Response 201:**
```json
{
  "id": "key_01H...",
  "key": "rb_live_xyz789...",
  "name": "Production server",
  "created_at": "2026-05-14T12:34:56Z"
}
```

The raw `key` is returned once and never again.

**Errors:**
- `401 unauthorized`
- `400 invalid_name`
- `404 auth_disabled`

### `GET /auth/keys`

List the caller's API keys. Raw keys are never returned.

**Response 200:**
```json
{
  "keys": [
    {
      "id": "key_01H...",
      "name": "Default",
      "created_at": "2026-05-14T12:34:56Z",
      "last_used_at": "2026-05-14T13:45:00Z",
      "revoked_at": null
    }
  ]
}
```

### `DELETE /auth/keys/{id}`

Revoke a key. Sets `revoked_at`; the row is kept for audit.

**Response 204** (no body).

**Errors:** `404 not_found` if the key doesn't belong to the caller;
`404 auth_disabled` if auth is off.

### `GET /auth/usage`

Returns current-period usage and quotas.

When `RB_ENABLE_QUOTAS` is unset/false (the OSS default) or when
`RB_REQUIRE_AUTH` is unset/false, the response is the honest
`{"enabled": false}` envelope. The full shape below is only returned when
both auth and quotas are enabled.

**Response 200 (quotas enabled):**
```json
{
  "vectors_used": 12500,
  "vector_quota": 100000,
  "queries_today": 342,
  "daily_query_quota": 10000,
  "queries_reset_at": "2026-05-15"
}
```

`queries_reset_at` is a `YYYY-MM-DD` date string — the day the daily counter
was last reset.

## Datasets

### `POST /v1/datasets`

Create an empty dataset bound to the caller's tenant. Vectors are added later
via `POST /v1/datasets/{name}/vectors`.

**Request:**
```json
{ "name": "products", "dimension": 768 }
```

`name`: 1-64 chars, `[a-z0-9_-]+`, unique per tenant.
`dimension`: positive int, must match every vector posted to this dataset.

**Response 201:**
```json
{
  "name": "products",
  "dimension": 768,
  "status": "empty",
  "row_count": 0,
  "created_at": "2026-05-14T12:34:56Z"
}
```

**Status values:**
- `empty` — created, no vectors uploaded yet
- `validating` — validator has received records
- `indexing` — index_builder is building a shard
- `indexed` — at least one shard is queryable
- `error` — last build/validate failed (`error_message` populated)

**Errors:**
- `400 invalid_name`, `400 invalid_dimension`
- `409 dataset_exists`
- `401 unauthorized`

### `POST /v1/datasets/{name}/vectors`

Stream NDJSON records into the dataset.

**Content-Type:** `application/x-ndjson`

**Request body:** One JSON object per line:
```
{"id": "doc-1", "values": [0.1, 0.2, ...], "metadata": {"title": "A"}}
{"id": "doc-2", "values": [0.3, 0.4, ...], "metadata": {"title": "B"}}
```

Each record:
- `id`: string, 1-256 chars, caller-chosen
- `values`: array of floats, length must equal the dataset's `dimension`
- `metadata`: optional object, arbitrary JSON

**Response 202:**
```json
{
  "accepted": 1000,
  "rejected": 2,
  "errors": [
    { "line": 47, "reason": "dimension mismatch: got 767 expected 768" },
    { "line": 803, "reason": "values must be list[float]" }
  ],
  "job_id": "job_01H..."
}
```

`accepted` records are queued; `status` on the dataset moves to `validating`
then `indexing` then `indexed`. Poll via `GET /v1/datasets/{name}`.

`accepted` counts NDJSON lines that passed validation, counted *before*
upsert dedup — so if one request re-sends the same `id` twice, both lines
are counted in `accepted` even though last-write-wins collapses them to a
single stored row.

Uploading to a dataset that already has an index is processed incrementally
— only the new batch is indexed, not the whole dataset. The endpoint is an
**upsert**: a record whose `id` already exists in the dataset (or appears
twice within one request) overwrites the previous value and metadata — last
write wins — rather than producing a duplicate. See
[`docs/indexing.md`](../indexing.md) for the incremental-indexing behaviour.

**Errors:**
- `404 dataset_not_found`
- `400 invalid_ndjson` — body isn't NDJSON
- `429 vector_quota_exceeded` — `details.limit` and `details.used` populated
  (only when `RB_ENABLE_QUOTAS` is on)
- `413 payload_too_large` — body exceeds the 10 MiB per-request cap

### Bulk import — `POST /v1/datasets/{name}/imports*`

For large embedding dumps that exceed the `POST .../vectors` 10 MiB cap, use
the **async import-job** flow: the client stages the file directly into
object storage via a presigned upload, then a job validates and indexes it
asynchronously. Endpoints:

- `POST   /v1/datasets/{name}/imports` — create a job, returns a presigned upload target
- `POST   /v1/datasets/{name}/imports/{import_id}/complete` — signal upload done
- `GET    /v1/datasets/{name}/imports/{import_id}` — job status
- `GET    /v1/datasets/{name}/imports` — list a dataset's jobs

Supports NDJSON and Parquet uploads, `continue`/`abort` error modes, a
rejected-records file, and two-stage quota enforcement. The full contract is
in [`imports.md`](./imports.md).

### `GET /v1/datasets`

List the caller's datasets.

**Response 200:**
```json
{
  "datasets": [
    {
      "name": "products",
      "dimension": 768,
      "status": "indexed",
      "row_count": 12500,
      "created_at": "2026-05-14T12:34:56Z",
      "last_indexed_at": "2026-05-14T12:40:00Z",
      "error_message": null
    }
  ]
}
```

### `GET /v1/datasets/{name}`

Same per-dataset shape as a list entry.

**Errors:** `404 dataset_not_found`

### `DELETE /v1/datasets/{name}`

Soft-delete in catalog (sets `deleted_at`); shard catalog rows are purged in
the same transaction so a same-name re-create starts clean. Object-storage
bytes are reclaimed by a background sweep. Subsequent `GET` returns 404.

**Response 204**

## Query

### `POST /v1/query`

Vector similarity search. On the Control Plane this endpoint authenticates +
rate-limits + consumes daily quota, then reverse-proxies to a private Query
Data Plane node which runs the FAISS search. The customer-observable contract
is the same whether the result comes from the hot cache or the ephemeral
runner; the CP/DP split is invisible. Implementation:
`services/query_api/query_proxy.py` (CP side),
`services/query_api/dp_query.py` + `services/query_api/v1_query.py` (DP side).

**Request:**
```json
{
  "dataset": "products",
  "vector": [0.1, 0.2, ...],
  "top_k": 10,
  "filter": { "category": "books", "year": 2024 }
}
```

- `dataset` (string, required): caller's dataset name
- `vector` (array of floats, required): length must equal the dataset's
  `dimension`
- `top_k` (int, optional, default 10, max 1000)
- `filter` (object, optional): metadata filter. A **flat** object of
  `field -> value` with **AND-of-equals** semantics. A result is included
  only if, for **every** key in `filter`, the record's `metadata` contains
  that key with an **exactly-equal** value.
  - A record missing a filtered key is **excluded**.
  - Comparison is **strict and not coerced**: string compares to string,
    number to number. A type mismatch does not match — e.g. `filter` value
    `"2024"` (string) does **not** match metadata `2024` (number).
  - When `filter` is absent, `null`, or `{}`, no filtering is applied.
  - A `null` filter value (e.g. `{ "filter": { "category": null } }`) is
    accepted by validation but **never matches** any record — `null` is
    not a meaningful equality target in v1, so every record carrying that
    key is excluded. The query still succeeds with HTTP 200 (typically
    `"matches": []`).
  - No ranges, OR, or nesting in v1. A nested object or array as a filter
    value is rejected with `400 invalid_request`.
- `nprobe` (int, optional): per-query IVF `nprobe` override. Server default
  is `RB_QUERY_NPROBE` (64). Ignored for filtered queries — they are
  exhaustive by construction. See
  [`docs/indexing.md`](../indexing.md) for the recall/latency tradeoff.

**Partial results:** filtering happens *after* the nearest-neighbour search
(FAISS cannot filter by metadata). A filtered query is run **exhaustively**
— every IVF cell is scanned and the AND-of-equals predicate is applied to
every candidate — so a highly selective filter can legitimately return
**fewer than `top_k`** matches; that is an exact answer, not an
approximation or an error. A filter matching nothing returns
`"matches": []` with HTTP 200. The implementation is
`services/query_api/v1_query.py:metadata_matches_filter` +
`_run_faiss_search`.

**Response 200:**
```json
{
  "matches": [
    { "id": "doc-42", "score": 0.0,   "metadata": { "title": "A" } },
    { "id": "doc-7",  "score": 0.118, "metadata": { "title": "B" } }
  ],
  "latency_ms": 23,
  "mode": "hot"
}
```

`score` is the raw FAISS **L2 (squared Euclidean) distance** — **lower is
closer**, `0.0` is an exact match. `matches` is sorted nearest-first
(ascending score). It is a distance, not a normalised similarity.

`mode` is one of:
- `"hot"` — served from the in-process FAISS shard cache.
- `"cold"` — served from FAISS, but the shard was loaded into the cache for
  the first time on this request.
- `"ephemeral"` — the dataset has no shard yet (not `indexed`). The query
  is enqueued on the ephemeral runner; the immediate response carries
  `matches: []` plus a `job_id` to poll.

When `mode == "ephemeral"`:
```json
{ "matches": [], "latency_ms": 12, "mode": "ephemeral", "job_id": "job_01H..." }
```

Then poll:

### `GET /v1/query/status/{job_id}`

**Response 200:**
```json
{ "ready": true, "matches": [...], "latency_ms": 142, "mode": "ephemeral" }
```

or `{ "ready": false }` if still computing (also covers an unknown
`job_id`).

**Errors:**
- `404 dataset_not_found` (also covers cross-tenant — never 403)
- `400 dimension_mismatch` — `details.expected`, `details.got`
- `400 top_k_out_of_range`
- `400 nprobe_out_of_range`
- `429 query_quota_exceeded` — `details.limit`, `details.reset_at` (only
  when quotas are enabled)
- `429 rate_limited` — `details.limit_rps`, `details.burst` (only when
  quotas are enabled)
- `502 bad_gateway`, `503 query_unavailable`, `504 query_timeout` — the
  CP→DP hop failed.
- `503 cache_unavailable`, `503 storage_unavailable`, `503 ephemeral_error`
  — the search itself could not run (shard cache fs is unreadable, the
  object store is unreachable, or the ephemeral runner errored). A
  successful HTTP 200 from this endpoint always implies the `matches`
  array is the actual top-K result — never a silent empty list standing
  in for a failed search. The same set of codes is also surfaced by
  `GET /v1/query/status/{job_id}` when an ephemeral worker fails after
  the job was enqueued (the status poll returns 503 with the envelope
  instead of `{"ready": true, "matches": []}`).

## Error code reference

| HTTP | code | meaning |
|---|---|---|
| 400 | `invalid_request` | request body malformed or invalid shape |
| 400 | `invalid_email` | signup email failed validation |
| 400 | `weak_password` | password too short |
| 400 | `invalid_name` | dataset or key name failed validation |
| 400 | `invalid_dimension` | dimension non-positive |
| 400 | `invalid_ndjson` | upload body malformed |
| 400 | `invalid_format` | import `format` not `ndjson`/`parquet` |
| 400 | `invalid_error_mode` | import `error_mode` not `continue`/`abort` |
| 400 | `upload_missing` | import `complete` called before the file was staged |
| 400 | `dimension_mismatch` | query vector length != dataset dimension |
| 400 | `top_k_out_of_range` | top_k < 1 or > 1000 |
| 400 | `nprobe_out_of_range` | nprobe < 1 or > 1024 |
| 401 | `unauthorized` | missing/invalid/expired token |
| 401 | `invalid_credentials` | login failed |
| 404 | `auth_disabled` | `/auth/*` mutation route when `RB_REQUIRE_AUTH` is off |
| 404 | `not_found` | API key not owned by caller |
| 404 | `dataset_not_found` | dataset missing OR not owned by caller |
| 404 | `import_not_found` | import job missing OR not owned by caller |
| 409 | `email_taken` | signup conflict |
| 409 | `dataset_exists` | dataset name already used by this tenant |
| 409 | `import_not_pending` | import `complete` called on a non-`awaiting_upload` job |
| 413 | `payload_too_large` | ingest body > 10 MiB |
| 429 | `vector_quota_exceeded` | per-tenant vector cap hit (quotas enabled) |
| 429 | `query_quota_exceeded` | daily query cap hit (quotas enabled) |
| 429 | `rate_limited` | per-key token bucket exhausted (quotas enabled) |
| 500 | `internal_error` | catch-all |
| 502 | `bad_gateway` | CP→DP transport/protocol error mid-response |
| 503 | `query_unavailable` | CP→DP connect failure after retries |
| 503 | `service_unavailable` | Postgres pool exhausted |
| 503 | `cache_unavailable` | shard cache filesystem unreadable / unwritable |
| 503 | `storage_unavailable` | object-store fetch failed (S3 outage, missing shard) |
| 503 | `ephemeral_error` | ephemeral runner failed for an unclassified reason |
| 503 | `cache_capacity_exceeded` | SSD shard tier rejected the load (admission floor); raise `RB_SHARD_TIER_BYTES` |
| 504 | `query_timeout` | CP→DP read timeout |

## Rate limits and quotas

When `RB_ENABLE_QUOTAS` is unset/false (the OSS default), every quota and
rate-limit below is a **no-op** — no token-bucket math, no per-tenant
counters incremented, `/auth/usage` returns `{"enabled": false}`. Set
`RB_ENABLE_QUOTAS=true` to enable quota enforcement; the defaults below apply.
Implementation: `services/auth/quota.py`.

- **Per API key:** 50 req/s sustained, burst 100 (in-memory token bucket;
  reset across process restarts).
- **Per tenant per day:** 10,000 queries per tenant per day (default).
- **Per tenant total:** 100,000 vectors per tenant (default).

Defaults are overridable via `RB_RATE_LIMIT_RPS`, `RB_RATE_LIMIT_BURST`,
`RB_TEST_QUERY_QUOTA`, `RB_TEST_VECTOR_QUOTA` — see
[`quotas.md`](./quotas.md).

## Versioning

This is the `v1` contract. Breaking changes ship as `/v2`. Additive changes
(new fields, new endpoints) ship in-place without a version bump.
